Content Broadcasting System - Architecture Notes

1. Authentication & RBAC Flow
- The backend uses JWT-based authentication for all private APIs.
- Login returns a signed access token containing user id, email and role.
- An authentication middleware validates the Bearer token, verifies the JWT, loads the user from the database and attaches a sanitized user object to req.user.
- An authorization middleware enforces role-based access using allowed role arrays.
- Teacher-only operations: upload content, edit own content, delete own content, resubmit rejected content.
- Principal-only operations: moderate content status, create and manage schedules.
- Public broadcasting API does not require authentication.

2. Subject-Based System Design
- Every content item belongs to a subject such as Maths or Science.
- A ContentSlot row represents a unique subject channel.
- During teacher upload/update, the system ensures the subject exists in the ContentSlot table.
- Scheduling is slot-based, so each subject can have an independent broadcast rotation.
- Public broadcasting can optionally filter by subject, and rotation is calculated separately for each subject.

3. Upload Handling Approach
- Multer is used for local file storage in the backend uploads directory.
- Supported formats are JPG, PNG and GIF.
- File size is limited to 10MB.
- Upload validation is handled before controller logic and standardized into the common Result response format.
- Teacher upload stores file path, mime type and size in the Content table.

4. Approval Workflow Design
- Teacher uploads content with title, subject, optional description, start time, end time and optional rotation duration.
- Uploaded content is stored in pending state and is not public immediately.
- Principal reviews content using retrieval APIs and updates status using a dedicated status update API.
- If approved, the system stores approved_by and approved_at.
- If rejected, the system stores rejection_reason and clears approval metadata.
- Rejected content can be edited by the teacher but remains rejected until the teacher explicitly calls the resubmit API, which changes status back to pending.

5. Scheduling / Rotation Logic
- Scheduling metadata is stored in ContentSchedule using content_id, slot_id, rotation_order and duration.
- Only approved content can be added to a schedule.
- The public live API loads scheduled rows for a given teacher, joins them with content and slot data, and keeps only approved content that is currently inside its start_time and end_time window.
- The filtered schedules are grouped by subject.
- Within each subject group, schedules are ordered by rotation_order and duration is used to compute a looping cycle.
- The system calculates total loop duration for the subject, computes elapsed time from the earliest active start time and uses modulo arithmetic to determine the currently active scheduled item.
- The live API returns one active item per subject for the requested teacher.
- If no approved or active content exists, the API returns an empty data response with the message "No content available".

6. Database Design Decisions
- Users table stores principal and teacher accounts with hashed passwords.
- Content table stores core content metadata, approval state, teacher-defined visibility window and optional rotation duration.
- ContentSlot stores unique subjects to support subject-based channels.
- ContentSchedule stores ordered rotation configuration per slot.
- Associations:
	- User -> Content through uploaded_by and approved_by
	- Content -> ContentSchedule through content_id
	- ContentSlot -> ContentSchedule through slot_id
- This design keeps upload, approval and broadcast scheduling concerns separated while still being simple enough for the assignment.

7. Folder Structure
- src/routes: HTTP route definitions and request data extraction.
- src/controllers: request validation, response shaping and orchestration.
- src/services: business logic and database interaction.
- src/middlewares: authentication, authorization and upload handling.
- src/models: Sequelize models and associations.
- src/utils: shared constants, response object and validation helpers.
- Separate schedule module files are used because scheduling is a distinct business area from content CRUD.

8. Middleware Usage
- authentication.js validates JWT and loads the current user.
- authorization.js restricts APIs by allowed roles.
- upload.js handles local file upload validation and storage.
- Route-level upload wrapper converts Multer errors into the common response structure.

9. Scalability Approach
- The current implementation is intentionally simple and assignment-focused, but it is designed so future improvements are possible.
- The /content/live API is a good candidate for Redis caching because it is public and read-heavy.
- Rate limiting is implemented on the public live API to reduce abuse risk and can be tuned further if traffic grows.
- Local upload storage can be replaced with S3 or another object store without changing the content lifecycle design.
- Broadcast calculation can later be moved to a background or cached layer if traffic grows.
- Schedule and content modules are separated enough to support future analytics, audit logs or queue-based processing.

10. Assumptions
- Content is created directly in pending state instead of persisting an intermediate uploaded state transition.
- Public live response returns active content per subject for a teacher, which aligns with the subject-based rotation requirement.
- Schedule creation is principal-driven and only approved content can be scheduled.
